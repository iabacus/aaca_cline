# 문서 작성 가이드

## 작성 방법

기술적인 내용을 작성할 때는 기술의 세부 사항보다는 새로운 것을 배우거나 해결책을 얻는 것에 중점을 둡니다.

## 매력적인 제목 작성

효과적인 제목은 다음을 수행해야 합니다:

- 감정적 반응 유발
- 누군가의 목표 강조
- 꿈이나 열망 제공
- 믿음에 도전하거나 논평
- 누군가의 문제 해결

"How I do something" 또는 "How to do something" 대신 "How you can do something"으로 독자의 관점에서 프레임을 구성하세요.

## 형용사 사용 금지

형용사를 피하고 대신 증거를 사용하세요. "production ready"라고 말하는 대신 "100개 서버 또는 초당 100만 문서로 확장"과 같이 작성할 수 있습니다.

"blazingly fast"와 같은 말 대신 "200배 빠름" 또는 "30% 빠름"이라고 말하세요.

진술이 다음과 같은지 추적하세요:
- 시각화 가능
- 거짓으로 증명 가능
- 당신만이 말할 수 있음

## 소화하기 쉽게 유지
- 5분 읽기를 목표로 하세요
- 10학년 읽기 수준으로 작성
- 긴 단락 나누기
- 헤더와 글머리 기호 사용

## 스캔 가능하게 만들기
- 핵심 포인트를 굵게 표시
- 3-4개 단락마다 부제목 사용
- 충분한 여백 포함
- 관련 예제 추가

# Cline 문서 작성 가이드

## 기능 설명을 위한 일반 원칙

기능에 대해 이야기할 때는 간단한 용어로 기능이 무엇인지 다루는 사람이 읽을 수 있는 설명으로 시작하는 것이 좋습니다. 전문 용어를 건너뛰고 이전에 본 적이 없는 사람에게 이야기하는 것처럼 설명하세요.

위치와 사용법을 하나의 흐르는 섹션으로 결합하세요. 사용자에게 기능을 찾을 수 있는 위치와 사용 방법을 정확히 알려주되, 지침을 글머리 기호, 번호 매기기 목록, 코드 예제, mintlify 컴포넌트 및 헤더/부제목의 적절한 균형으로 자연스러운 산문으로 엮으세요.

실제 파일, 워크플로우 또는 코드와 같은 실제 예제로 기능을 실제로 보여주세요. 사용자는 추상적인 설명이 아닌 구체적인 구현을 봐야 합니다.

기능에 대해 이야기할 때 상상력을 자극하는 영감 섹션을 포함하세요. 이 섹션은 사람들을 이해에서 행동으로 밀어냅니다.

## 실제로 작동하는 작성 원칙

### 이해가 아닌 행동을 위해 작성

문서는 사용자가 시도하도록 동기를 부여해야 합니다. 무언가가 어떻게 작동하는지 설명하는 대신 사용자가 그것으로 무엇을 달성할 수 있는지에 집중하세요.

### 자연스러운 스토리 흐름 만들기

"이것은 무엇인가?"에서 "어떻게 사용하나?"로, "실제 예제는 다음과 같습니다"로, "이것으로 무엇을 할 수 있는지 상상해보세요"로 자연스럽게 진행되는 대화처럼 느껴져야 합니다.

### 장난감 데모가 아닌 실제 예제 표시

사용자가 복사하고 적응할 수 있는 실제 워크플로우 파일, 실제 코드 스니펫 및 구체적인 구현을 제공하세요.

### 스캔 가능하지만 단편화되지 않게 유지

완전히 읽을 때 자연스럽게 흐르는 산문으로 작성하되, 사용자가 문제 해결 시 특정 정보를 빠르게 찾을 수 있도록 구조화하세요.

## 언어 및 톤 가이드라인

명확하게 작성하되 단순화하지 마세요. 가능한 경우 간단한 언어를 사용하되, 사용자가 알아야 하는 기술 용어를 피하지 마세요.

작성을 대화적이고 격려적으로 만드세요. "you can also try" 또는 "when that works"와 같은 문구는 엄격한 지시 언어보다 더 자연스럽게 느껴집니다.

콘텐츠를 간결하고 목적이 있게 유지하세요. 모든 문장은 사용자가 무언가를 이해하거나 무언가를 하는 데 도움이 되어야 합니다.

컨텍스트와 추론을 구축하세요. 사용자는 무엇을 해야 하는지뿐만 아니라 왜 하는지 이해하고 싶어합니다.

## 나쁜 예

**굵은 텍스트** 콜론 그리고 더 많은 텍스트의 글머리 기호 패턴을 싫어합니다:

```
#### macOS

1. **bash로 전환**: Cline 설정 → 터미널 → 기본 터미널 프로필 → "bash" 선택
2. **Oh-My-Zsh 일시적으로 비활성화**: zsh를 사용하는 경우 `mv ~/.zshrc ~/.zshrc.backup`을 시도하고 VSCode를 다시 시작
```

대신 자연스럽게 읽히는 아름다운 문서를 작성해야 합니다:

```
#### macOS

가장 일반적인 수정은 bash로 전환하는 것입니다. Cline 설정 → 터미널 → 기본 터미널 프로필로 이동하여 드롭다운에서 "bash"를 선택하세요.

여전히 문제가 있는 경우 Oh-My-Zsh가 터미널 통합을 방해할 수 있습니다. 일시적으로 비활성화해 보세요:
- `mv ~/.zshrc ~/.zshrc.backup` 실행
- VSCode 다시 시작
```

## Mintlify 컴포넌트 관용적으로 사용

### 프레임이 있는 시각적 콘텐츠

비디오와 이미지는 원시 HTML이나 markdown을 사용하는 대신 `<Frame>` 컴포넌트로 래핑해야 합니다.

```jsx
<Frame>
	<iframe
		style={{ width: "100%", aspectRatio: "16/9" }}
		src="https://www.youtube.com/embed/your-video-id"
		title="기능 데모"
		frameBorder="0"
		allow="accelerometer; autoplay; clipboard-write; encrypted-media; gyroscope; picture-in-picture; web-share"
		allowFullScreen
	/>
</Frame>
```

### 탐색 및 개요를 위한 카드

카드는 상세한 문서에 연결되는 스캔 가능한 개요를 만드는 데 탁월합니다.

```jsx
<Columns cols={2}>
  <Card title="기능 이름" icon="relevant-icon" href="/link/to/docs">
    이 기능이 무엇을 하는지 그리고 누군가가 왜 사용할지 설명하는 간단한 설명.
  </Card>
</Columns>
```

### 컨텍스트를 위한 팁과 노트

`<Tip>` 컴포넌트를 사용하여 주요 콘텐츠를 어지럽히지 않고 향상시키는 유용한 정보를 제공하세요:

```jsx
<Tip>
	프로 팁: 단일 메시지에 여러 @ 멘션을 결합하여 Cline에게 
	문제에 대한 포괄적인 컨텍스트를 제공할 수 있습니다.
</Tip>
```

## 인간처럼 작성, AI처럼 작성하지 마세요

짧고 직접적으로 유지하세요. 더 적은 단어로 말할 수 있다면 그렇게 하세요.

일반적인 문장 길이를 사용하세요. 누군가와 이야기할 때처럼 자연스럽게 짧고 긴 문장을 섞으세요.

기업 용어를 제거하세요. "utilize" 대신 "use"라고 말하세요. "in order to" 대신 "to"라고 말하세요.

명백한 것을 과도하게 설명하지 마세요. 사용자가 문서를 읽고 있다면 기본 개념을 이해할 것입니다.

## em 대시나 이모지를 절대 사용하지 마세요

그게 전부입니다. 절대 사용하지 마세요. AI만 em 대시나 이모지로 작성합니다.

## Cline 의인화

Cline을 언급할 때 항상 "it"이 아닌 "him"이라고 부르세요.

## 워크플로우를 공유할 때 "I" 사용

개인적인 터치를 추가하는 것이 큰 도움이 됩니다. 개발자에서 개발자로 Cline을 개인적으로 어떻게 사용하는지 공유하기 위해 "I"를 사용하는 좋은 예가 문서에 있습니다.

## 관련 문서 페이지 교차 링크

문서 작성을 마치면 교차 링크를 만드세요. 관련 문서가 있으면 링크하세요.
