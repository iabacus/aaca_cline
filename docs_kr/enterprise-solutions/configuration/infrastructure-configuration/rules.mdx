규칙은 코딩 선호도, 표준, 모범 사례에 대한 가이드를 Cline에 제공하는 커스텀 지침 파일입니다. 작업 시 이 지침들이 Cline의 컨텍스트에 추가됩니다.

## 규칙이란?

규칙은 `.clinerules/` 디렉터리에 저장되는 간단한 마크다운 파일이며, 팀의 관례, 선호, 가이드라인을 담고 있습니다. Cline이 다음을 이해하도록 돕습니다:

- 코딩 스타일과 관례
- 선호하는 라이브러리와 프레임워크
- 아키텍처 패턴
- 테스트 전략
- 문서화 표준
- 커뮤니케이션 선호도

<Tip>
규칙은 그냥 `.md` 파일입니다. 복잡한 설정이 필요 없습니다!
</Tip>

## 빠른 예시

TypeScript 개발을 안내하는 간단한 규칙 파일 예시:

```markdown
# TypeScript Conventions

## Code Style
- Use 2-space indentation
- Prefer `const` over `let`
- Always use explicit return types for functions
- Use named exports instead of default exports

## Testing
- Write unit tests for all utility functions
- Use Vitest as the testing framework
- Aim for 80%+ code coverage

## Dependencies
- Prefer native TypeScript features over external libraries
- Use Zod for runtime type validation
- Use date-fns for date manipulation
```

## 규칙 만들기

<Tabs>
  <Tab title="/newrule 명령 사용">
    규칙을 만드는 가장 쉬운 방법은 `/newrule` 명령입니다:
    
    1. Cline과의 대화 중 `/newrule` 입력
    2. Cline이 대화 내용과 선호도를 분석
    3. `.clinerules/`에 적절한 이름의 `.md` 파일 생성
    
    **예시:**
    ```
    /newrule
    
    Based on our conversation, create a rule for React component structure
    ```
  </Tab>
  
  <Tab title="수동 생성">
    규칙 파일을 수동으로 만들 수도 있습니다:
    
    1. 저장소 루트에 `.clinerules/` 디렉터리 생성
    2. 가이드라인을 담은 마크다운 파일 추가
    3. `react-patterns.md`, `api-conventions.md`처럼 설명적인 이름 사용
    
    **파일 구조:**
    ```
    your-repo/
    ├── .clinerules/
    │   ├── typescript-style.md
    │   ├── testing-standards.md
    │   └── code-review-checklist.md
    └── src/
    ```
  </Tab>
</Tabs>

## 규칙 관리

### 규칙 토글

개별 규칙 파일을 활성/비활성화할 수 있습니다:

1. Cline 인터페이스의 규칙 아이콘 클릭
2. 필요한 규칙을 켜거나 끔
3. 변경 사항은 새 작업에 즉시 적용

<Note>
규칙을 비활성화해도 파일은 유지되며 언제든 다시 활성화할 수 있습니다.
</Note>

### 엔터프라이즈 원격 규칙

<Info>
엔터프라이즈 배포에서는 **원격 글로벌 규칙**을 구성하여 모든 팀원에게 적용할 수 있습니다. 이는 인프라 구성으로 관리되며 개별 개발자가 끌 수 없습니다.

원격 규칙에 대한 자세한 내용은 [자체 호스팅 구성](/enterprise-solutions/configuration/infrastructure-configuration/overview)을 참고하세요.
</Info>

## 호환 포맷

Cline은 다른 AI 코딩 도구의 규칙도 지원합니다:

| 파일/디렉터리 | 도구 | 위치 |
|----------------|------|----------|
| `.cursorrules` | Cursor | 워크스페이스 루트(단일 파일) |
| `.cursor/rules/` | Cursor | 워크스페이스 디렉터리(`.mdc` 파일) |
| `.windsurfrules` | Windsurf | 워크스페이스 루트(단일 파일) |
| `AGENTS.md` | Various | 워크스페이스 루트 + 재귀 검색 |

<Note>
**AGENTS.md 동작:** Cline은 워크스페이스 루트에 최상위 `AGENTS.md`가 있을 때만 하위 `AGENTS.md`를 재귀적으로 검색합니다. 발견되면 모든 `AGENTS.md`를 상대 경로 헤더와 함께 병합합니다.
</Note>

이 파일들은 `.clinerules/`와 동일하게 동작하며 독립적으로 켜고 끌 수 있습니다.

## 규칙 예시

<AccordionGroup>
  <Accordion title="API 설계 표준" icon="code">
```markdown
# API Design Standards

## REST Conventions
- Use plural nouns for endpoints (`/users`, not `/user`)
- Use HTTP methods semantically (GET, POST, PUT, DELETE)
- Return appropriate status codes

## Response Format
\`\`\`typescript
{
  data: T,
  error?: string,
  metadata?: {
    page: number,
    total: number
  }
}
\`\`\`

## Error Handling
- Always return error messages in `error` field
- Use 4xx for client errors, 5xx for server errors
- Include request ID in error responses
```
  </Accordion>
  
  <Accordion title="테스트 요구사항" icon="vial">
```markdown
# Testing Requirements

## Test Organization
- Place tests next to source files (`Button.test.tsx`)
- Use `describe` blocks to group related tests
- Write descriptive test names

## Coverage Requirements
- Unit tests for all utility functions
- Integration tests for API endpoints
- E2E tests for critical user flows
- Minimum 80% coverage for new code

## Mocking Strategy
- Mock external API calls
- Use test fixtures for complex data
- Prefer dependency injection for testability
```
