워크플로는 재사용 가능한 지침이 담긴 마크다운 파일이며, 슬래시 명령으로 필요할 때 호출할 수 있습니다. 항상 적용되는 가이드가 아니라 "필요할 때 호출하는 규칙"이라고 생각하면 됩니다.

## 워크플로란?

워크플로는 [규칙](/enterprise-solutions/configuration/infrastructure-configuration/rules)과 비슷하지만, 한 가지 중요한 차이가 있습니다:

<CardGroup cols={2}>
  <Card title="규칙" icon="book">
    **항상 활성**
    
    켜져 있으면 모든 작업에 자동 적용
    
    예: 코딩 표준, 스타일 가이드
  </Card>
  
  <Card title="워크플로" icon="diagram-project">
    **요청 시 호출**
    
    슬래시 명령을 사용할 때만 적용
    
    예: 배포 체크리스트, 리뷰 프로세스
  </Card>
</CardGroup>

<Tip>
워크플로는 그냥 마크다운 파일입니다. 복잡한 설정이 필요 없습니다!
</Tip>

## 빠른 예시

간단한 배포 워크플로 예시:

**파일:** `.clinerules/workflows/deploy.md`

```markdown
# Deployment Workflow

Before deploying to production, ensure:

## Pre-Deployment Checklist
1. All tests passing (unit, integration, e2e)
2. Code review approved by 2+ engineers
3. Staging environment tested successfully
4. Database migrations reviewed
5. Rollback plan documented

## Deployment Steps
1. Create deployment branch from main
2. Run final test suite
3. Deploy to production
4. Monitor error rates for 30 minutes
5. Verify key user flows

## Post-Deployment
1. Update deployment log
2. Notify team in #deployments channel
3. Monitor metrics for 24 hours
```

**사용법:**
```
/deploy

I'm ready to deploy the new authentication feature
```

호출하면 해당 작업에만 워크플로 지침이 Cline 컨텍스트에 추가됩니다.

## 워크플로 만들기

<Tabs>
  <Tab title="수동 생성">
    `.clinerules/workflows/` 디렉터리에 워크플로 파일을 만듭니다:
    
    1. 저장소 루트에 `.clinerules/workflows/` 생성
    2. 워크플로 지침이 담긴 마크다운 파일 추가
    3. 슬래시 명령과 일치하는 설명적인 이름 사용
    
    **파일 구조:**
    ```
    your-repo/
    ├── .clinerules/
    │   └── workflows/
    │       ├── deploy.md
    │       ├── code-review.md
    │       └── bug-triage.md
    └── src/
    ```
  </Tab>
  
  <Tab title="슬래시 명령">
    대화 중에도 워크플로를 만들 수 있습니다:
    
    1. 정리하고 싶은 프로세스에 대해 대화
    2. `/newrule`을 입력하고 워크플로임을 명시
    3. Cline이 `.clinerules/workflows/`에 워크플로 파일 생성
    
    <Note>
    `/newrule` 명령은 규칙과 워크플로 모두 생성할 수 있습니다. 의도를 명확히 설명하세요.
    </Note>
  </Tab>
</Tabs>

## 워크플로 사용하기

### 워크플로 호출

`/` 뒤에 워크플로 파일명(확장자 `.md` 제외)을 입력하세요:

```
/deploy
/code-review  
/bug-triage
```

워크플로 지침은 현재 작업에만 추가됩니다.

### 워크플로 이름 규칙

- 소문자 + 하이픈 사용: `deploy.md`, `code-review.md`
- 짧고 기억하기 쉬운 이름
- 워크플로 목적을 나타내는 이름

<Warning>
워크플로 파일명은 슬래시 명령이 되므로, 입력과 기억이 쉬운 이름을 선택하세요.
</Warning>

## 글로벌 vs 워크스페이스 워크플로

<CardGroup cols={2}>
  <Card title="워크스페이스 워크플로" icon="folder">
    **위치:** 저장소의 `.clinerules/workflows/`
    
    **범위:** 해당 프로젝트에만 적용
    
    **용도:** 프로젝트별 프로세스와 체크리스트
  </Card>
  
  <Card title="글로벌 워크플로" icon="globe">
    **위치:** `Documents/Cline/Workflows/` 디렉터리
    
    **범위:** 모든 프로젝트에 적용
    
    **용도:** 어디서나 사용하는 개인 워크플로
  </Card>
</CardGroup>

<Info>
**우선순위:** 동일한 이름일 경우 로컬 워크플로가 글로벌 워크플로를 덮어씁니다.
</Info>

## 워크플로 관리

### 워크플로 토글

워크플로를 활성/비활성화할 수 있습니다:

1. Cline 인터페이스의 규칙 아이콘 클릭
2. "Workflows" 탭으로 이동
3. 필요한 워크플로를 켜고 끔

<Note>
워크플로를 비활성화하면 호출되지 않지만 파일은 유지됩니다. 다시 활성화하기 전까지 슬래시 명령이 동작하지 않습니다.
</Note>

### 엔터프라이즈 원격 워크플로

<Info>
엔터프라이즈 배포에서는 모든 팀원이 사용할 수 있는 **원격 글로벌 워크플로**를 구성할 수 있습니다. 이는 인프라 구성을 통해 관리됩니다.

원격 워크플로에 대한 자세한 내용은 [자체 호스팅 구성](/enterprise-solutions/configuration/infrastructure-configuration/overview)을 참고하세요.
</Info>

## 워크플로 예시

<AccordionGroup>
  <Accordion title="코드 리뷰 워크플로" icon="code-review">
```markdown
# Code Review Workflow

## Pre-Review Checklist
- [ ] Code follows project style guide
- [ ] All tests pass locally
- [ ] No console.log or debugging code
- [ ] Comments explain "why" not "what"
- [ ] PR description is clear and complete

## Review Focus Areas
1. **Architecture**: Does this fit our existing patterns?
2. **Security**: Any potential vulnerabilities?
3. **Performance**: Any obvious bottlenecks?
4. **Testing**: Are edge cases covered?
5. **Documentation**: Is it clear how to use new features?

## Review Response
- Address all feedback within 24 hours
- Mark conversations as resolved when addressed
- Re-request review after major changes
```
  </Accordion>
  
  <Accordion title="버그 분류 워크플로" icon="bug">
```markdown
# Bug Triage Workflow

## Information Gathering
1. Reproduce the bug in local environment
2. Identify affected versions/environments
3. Check if similar issues exist
4. Gather error logs and stack traces

## Priority Assessment
**P0 (Critical)**: Production down, data loss, security breach
**P1 (High)**: Major feature broken, significant user impact
**P2 (Medium)**: Minor feature broken, workaround available
**P3 (Low)**: Cosmetic issue, minimal impact

## Create Ticket
- Use template: "Bug Report"
- Add reproduction steps
- Include screenshots/videos if applicable
- Tag with affected component
- Assign priority label

## Next Steps
- P0/P1: Immediate fix required
- P2: Schedule for current sprint
- P3: Add to backlog
```
  </Accordion>
  
  <Accordion title="기능 계획 워크플로" icon="lightbulb">
```markdown
# Feature Planning Workflow

## Requirements Gathering
1. Define the user problem we're solving
2. List success criteria (measurable)
3. Identify edge cases and constraints
4. Document technical dependencies

## Design Considerations
1. How does this fit existing architecture?
2. What data models are needed?
3. What API changes are required?
4. How will this impact performance?

## Implementation Plan
1. Break into smaller, shippable pieces
2. Identify which pieces can be done in parallel
3. Note any feature flags needed
4. Plan for backwards compatibility

## Testing Strategy
1. What unit tests are needed?
2. What integration tests are needed?
3. How will we test edge cases?
4. What manual testing is required?
```
  </Accordion>
</AccordionGroup>

## 모범 사례

<AccordionGroup>
  <Accordion title="워크플로를 실행 중심으로" icon="list-check">
    워크플로에는 **실행 가능한 단계**를 담으세요:
    - ✅ "Run `npm test` and verify all tests pass"
    - ❌ "Make sure testing is done properly"

    - 진행 상황이 명확함

    - PR에서 업데이트 리뷰
    - 프로세스 진화의 이력 유지
  </Accordion>
</AccordionGroup>

## 워크플로 vs 규칙: 언제 무엇을 쓸까

| 규칙을 쓸 때 | 워크플로를 쓸 때 |
|----------------|-------------------|
| 가이드가 모든 작업에 적용되어야 할 때 | 프로세스를 가끔 호출할 때 |
| 거의 변하지 않는 표준 | 특정 상황 체크리스트 |
| 항상 적용되는 코딩 관례 | 요청 시 배포 프로세스 |
| 일반적인 코딩 스타일 | 특정 리뷰 절차 |

**예시:**
- **규칙**: "TypeScript strict 모드와 명시적 반환 타입을 사용"
- **워크플로**: "프로덕션 배포 시 10단계를 따르기"
