<Note>
이 명령은 **VS Code**에서만 사용할 수 있습니다. 인라인 댓글이 포함된 diff 뷰는 VS Code의 네이티브 diff 기능이 필요합니다.
</Note>

`/explain-changes`는 어떤 git diff든 AI 설명을 생성하는 슬래시 명령입니다. 완료된 작업의 변경을 설명하는 [Explain Changes 버튼](/features/explain-changes)과 달리, 이 명령은 커밋, 브랜치, 태그, PR, 스테이징 변경, 작업 디렉터리 등 어떤 두 git 참조 사이의 변경도 설명할 수 있습니다.

## 요구사항

<Note>
`/explain-changes` 명령은 **git 저장소**가 필요합니다. git으로 초기화된 디렉터리에서 작업하고 있는지 확인하세요.
</Note>

PR 설명을 위해서는 [GitHub CLI(gh)](https://cli.github.com/)가 설치되고 인증되어 있어야 합니다. GitLab 머지 리퀘스트 설명을 위해서는 [GitLab CLI(glab)](https://gitlab.com/gitlab-org/cli)가 설치되고 인증되어 있어야 합니다.

이 명령은 git 참조를 직접 사용하므로, Explain Changes 버튼과 달리 체크포인트 활성화가 필요하지 않습니다.


## 명령 사용법

채팅 입력에서 `/explain-changes`를 입력하세요. Cline은 다음을 수행합니다:

1. git 히스토리를 분석해 어떤 변경이 존재하는지 파악
2. 관련 파일을 읽어 컨텍스트 수집
3. 비교할 git 참조를 결정
4. 인라인 설명이 스트리밍되는 diff 뷰 생성


## 동작 방식

`/explain-changes`를 사용하면 Cline이:

1. **컨텍스트 수집**: git 명령으로 저장소 상태 파악
2. **변경 식별**: 참조 간 변경 파일 파악
3. **관련 파일 읽기**: 더 나은 설명을 위한 컨텍스트 구성
4. **generate_explanation 호출**: diff 뷰 생성 및 설명 스트리밍
5. **결과 표시**: 인라인 댓글이 있는 다중 파일 diff 열기

## 사용 사례

### 마지막 커밋 설명

가장 흔한 사용 사례 - 최신 커밋의 변경을 이해:

```
/explain-changes
```

Cline은 HEAD를 HEAD~1과 비교해 해당 커밋의 모든 변경을 설명합니다.

**언제 사용하나요:**
- 동료의 변경을 pull한 직후
- push 전 자신의 작업 리뷰
- merge 커밋이 가져온 변경 파악

### 커밋되지 않은 변경 설명

커밋 전에 진행 중인 변경을 이해:

```
/explain-changes for my uncommitted work
```

Cline이 HEAD와 작업 디렉터리를 비교해 수정된 파일을 설명합니다.

**언제 사용하나요:**
- 스테이징 전 변경이 완전한지 확인
- 긴 코딩 세션 후 변경 내용 회상
- 커밋 전 변경 사항 검증

### 스테이징된 변경 설명

커밋하려는 내용을 정확히 검토:

```
/explain-changes for my staged changes
```

Cline이 `git add`로 스테이징한 변경만 확인합니다.

**언제 사용하나요:**
- 커밋 전 अंतिम 리뷰
- 일부 변경만 스테이징했을 때 확인
- 의도치 않은 파일이 포함됐는지 검증

### 특정 커밋 설명

히스토리의 특정 커밋을 이해:

```
/explain-changes for commit abc123
```

또는 커밋 메시지로:

```
/explain-changes for the commit that added authentication
```

Cline이 해당 커밋을 찾아 변경을 설명합니다.

**언제 사용하나요:**
- 버그가 도입된 시점 조사
- 과거 의사결정 이해
- 기능이 구현된 방식 학습

### 커밋 범위 설명

여러 커밋을 한 번에 이해:

```
/explain-changes for the last 3 commits
```

또는 특정 범위:

```
/explain-changes from v1.0.0 to v1.1.0
```

Cline이 범위의 시작과 끝을 비교해 모든 변경을 설명합니다.

**언제 사용하나요:**
- 릴리스 변경 사항 파악
- 연관된 커밋 시리즈 리뷰
- 프로젝트 복귀 후 변경 사항 빠르게 파악

### Pull Request 설명

PR에 대한 AI 설명을 얻기:

```
/explain-changes for PR #42
```

Cline이 GitHub CLI로 PR을 가져와 변경을 설명합니다.

**언제 사용하나요:**
- 다른 사람의 PR 리뷰
- 승인 전 변경 이해
- 오픈 소스 PR에서 학습
- PR 피드백 준비

### 브랜치 차이 설명

두 브랜치 비교:

```
/explain-changes between main and feature-branch
```

또는 feature 브랜치의 변경만 보기:

```
/explain-changes for everything on my-feature that's not in main
```

**언제 사용하나요:**
- feature 브랜치 머지 전 검토
- 브랜치 간 분기 파악
- merge/rebase 전략 수립
- 동료 작업 내용 파악

### 특정 파일 변경 설명

특정 파일/디렉터리에 집중:

```
/explain-changes for src/auth in the last 5 commits
```

Cline이 diff를 필터링해 관련 변경만 표시합니다.

**언제 사용하나요:**
- 특정 모듈 변경 파악
- 중요한 파일 수정 추적
- 특정 기능의 진화 과정 학습

### 태그 이후 변경 설명

릴리스 이후 변경 파악:

```
/explain-changes since v2.0.0
```

Cline이 태그와 HEAD를 비교해 이후 변경을 설명합니다.

**언제 사용하나요:**
- 릴리스 노트 준비
- 배포 이후 변경 파악
- 변경 로그 정리

### 머지 커밋 설명

머지가 가져온 변경 파악:

```
/explain-changes for the merge from feature-x
```

Cline이 머지된 변경을 모두 설명합니다.

**언제 사용하나요:**
- 큰 feature 브랜치 머지 직후
- 머지 충돌 해결 내용 파악
- main에 합쳐진 변경 검토

### 스태시 변경 설명

스태시에 있는 변경을 검토:

```
/explain-changes for my stashed changes
```

Cline이 stash@{0}의 내용을 설명합니다.

**언제 사용하나요:**
- 스태시 적용 전 확인
- 스태시 유지/삭제 결정
- 며칠 전 스태시한 내용 회상

## 인터랙티브 댓글

[Explain Changes](/features/explain-changes) 버튼과 동일하게, 생성된 댓글은 완전히 인터랙티브합니다:

### 댓글에 답하기

어떤 댓글 스레드에서든 후속 질문을 할 수 있습니다:

- "왜 이 함수가 리팩터링되었나요?"
- "이 새 파라미터의 목적은 무엇인가요?"
- "이 변경이 breaking change를 일으킬 수 있나요?"
- "이 변경은 하위 호환이 되나요?"

AI는 코드와 전체 변경을 모두 이해한 상태에서 맥락 있는 답변을 제공합니다.

### 메인 채팅으로 이동

댓글 스레드 제목 영역을 클릭하면 해당 대화를 Cline의 메인 채팅으로 옮길 수 있습니다. 다음 상황에서 유용합니다:

- Cline이 추가 변경을 하길 원할 때
- 논의 중 추가 조사가 필요하다고 드러났을 때
- Cline의 전체 기능으로 계속 작업하고 싶을 때
- 리뷰 댓글이 개선 아이디어를 촉발했을 때

### generate_explanation 도구

내부적으로 `/explain-changes`는 다음 파라미터로 `generate_explanation` 도구를 사용합니다:

| 파라미터 | 설명 | 예시 |
|-----------|-------------|---------|
| `title` | diff 뷰의 설명 제목 | "Changes in commit abc123" |
| `from_ref` | 이전 상태의 Git 참조 | `HEAD~1`, `main`, `origin/main` |
| `to_ref` | 이후 상태의 Git 참조(옵션) | `HEAD`, `develop` |

`to_ref`를 생략하면 작업 디렉터리와 비교합니다.

## 더 나은 설명을 위한 팁

1. **구체적으로 말하기**: `/explain-changes`만 입력하는 것보다 "PR #42의 인증 변경을 설명해줘"처럼 요청하면 더 좋은 컨텍스트를 제공합니다.

2. **의도 질문하기**: AI는 무엇이 바뀌었는지뿐 아니라 왜 바뀌었는지도 설명할 수 있습니다. "이 변경이 어떤 문제를 해결했나요?"처럼 물어보세요.

3. **다른 명령과 연계**: 이슈를 조사한 뒤 `/explain-changes`로 변경을 이해하고, 이어서 Cline과 개선을 구현하세요.

4. **학습용으로 사용**: 새 코드베이스 온보딩 시, 중요한 PR이나 커밋에 `/explain-changes`를 사용해 기능이 어떻게 만들어졌는지 파악하세요.
