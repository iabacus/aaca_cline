Cline은 VS Code의 멀티 루트 워크스페이스를 지원해, 하나의 창에서 여러 프로젝트 폴더나 저장소를 관리할 수 있습니다. 모노레포든 별도 Git 저장소든, Cline은 모든 폴더에서 파일 읽기/쓰기와 명령 실행을 수행할 수 있습니다.

<Warning>
멀티 루트 워크스페이스에는 두 가지 제한이 있습니다:
- **Cline rules**는 기본 워크스페이스 폴더에서만 동작합니다
- **체크포인트**는 비활성화됩니다(단일 폴더로 돌아오면 복원)

자세한 내용은 [현재 제한사항](#current-limitations)을 참고하세요.
</Warning>

## 멀티 루트 워크스페이스 이해하기

시작하기 전에, 관련 프로젝트를 구성하는 두 가지 일반적인 패턴을 이해하면 도움이 됩니다.

### 왜 멀티 루트 워크스페이스를 사용할까?

Cline은 여러 프로젝트나 저장소에 걸친 작업을 수행할 수 있습니다:

- **리팩터링**: API 계약을 업데이트하고 여러 저장소의 소비 코드를 수정
- **기능 개발**: 프런트엔드, 백엔드, 공유 코드를 모두 건드리는 기능 구현
- **의존성 업데이트**: 관련 프로젝트 전반의 버전 업그레이드 조율
- **문서화**: 여러 저장소의 코드를 참조하는 문서 생성

**예시 프롬프트:**
```
contracts 저장소의 User 타입을 업데이트한 다음, frontend와 backend 모두 새 필드를 사용하도록 수정해줘. API가 새 필수 필드를 검증하도록 해줘.
```

## 멀티 루트 워크스페이스 설정

### 모노레포 vs 다중 저장소

**모노레포**: 하나의 Git 저장소에 여러 프로젝트/패키지가 포함됩니다. 모든 코드는 동일한 버전 히스토리를 공유합니다.

```
my-company/                    # 단일 Git 저장소
├── .git/
├── packages/
│   ├── web/                   # React 프런트엔드
│   ├── api/                   # Node.js 백엔드
│   └── shared/                # 공통 유틸리티
└── package.json
```

**다중 저장소**: 각각 독립적인 히스토리를 가진 여러 Git 저장소를 하나의 VS Code 워크스페이스로 함께 엽니다.

```
~/projects/
├── fullstack.code-workspace   # 워크스페이스 설정 파일
├── frontend/                  # git@github.com:acme/frontend.git
│   └── .git/
├── backend/                   # git@github.com:acme/backend.git
│   └── .git/
└── contracts/                 # git@github.com:acme/api-contracts.git
    └── .git/
```

Cline은 두 패턴 모두를 지원하며, 일부 폴더만 Git 저장소이고 나머지는 아닌 하이브리드 구성도 지원합니다. 핵심 차이는 다중 저장소에서는 각 폴더가 고유한 `.git` 디렉터리를 가지며 Cline이 이를 독립적으로 추적한다는 점입니다.

### 워크스페이스에 폴더 추가

다음 방법으로 폴더를 워크스페이스에 추가할 수 있습니다:

- **File 메뉴**: VS Code에서 `File > Add Folder to Workspace`
- **드래그 앤 드롭**: 폴더를 VS Code 파일 탐색기로 드래그
- **워크스페이스 파일**: `.code-workspace` 파일 생성(팀에 권장)
- **명령 팔레트**: `Workspaces: Add Folder to Workspace` 실행

자세한 설명은 [Microsoft의 멀티 루트 워크스페이스 가이드](https://code.visualstudio.com/docs/editor/multi-root-workspaces)를 참고하세요.

## 여러 저장소에서 작업하기

서로 다른 Git 저장소를 하나의 워크스페이스로 열면, Cline은 각 저장소를 고유한 프로젝트로 인식하고 독립적인 버전 관리를 수행합니다.

### 저장소별로 Cline이 추적하는 것

각 워크스페이스 폴더마다 Cline은 다음을 감지합니다:

| 속성 | 설명 |
|----------|-------------|
| **Path** | 폴더의 절대 경로 |
| **Name** | 폴더 이름 또는 워크스페이스 파일에서 유도 |
| **VCS Type** | Git, Mercurial 또는 없음 |
| **Commit Hash** | 현재 HEAD 커밋(Git/Mercurial 저장소) |

즉, 프런트엔드와 백엔드가 서로 다른 커밋, 다른 브랜치, 심지어 다른 VCS를 사용해도 Cline이 이를 이해합니다.

<Note>
Cline은 모든 워크스페이스 폴더의 VCS 정보를 감지하지만, 다음 기능은 **기본 워크스페이스**(첫 번째 폴더)만 사용합니다: [Cline rules](/features/cline-rules), [workflows](/features/slash-commands/workflows/index), 그리고 `@git` 멘션 같은 [Git 관련 기능](/features/at-mentions/git-mentions).
</Note>

## 워크스페이스 간 파일 참조

### 자연어 참조

Cline은 워크스페이스에 대한 자연어 참조를 이해합니다:

```
"frontend 폴더의 package.json을 읽어줘"
```

```
"backend의 user 모델과 contracts의 TypeScript 타입을 비교해줘"
```

```
"모든 워크스페이스에서 TODO 주석을 검색해줘"
```

### 워크스페이스 힌트 문법

명시적으로 참조하려면 `@workspace:path` 문법을 사용하세요:

| 문법 | 설명 |
|--------|-------------|
| `@frontend:src/App.tsx` | "frontend" 워크스페이스의 파일 |
| `@backend:server.ts` | "backend" 워크스페이스의 파일 |
| `@contracts:types/` | "contracts" 워크스페이스의 폴더 |

이 문법은 특히 다음 상황에서 유용합니다:
- 여러 워크스페이스에 같은 이름의 파일이 있을 때
- 어떤 프로젝트를 의미하는지 명확히 하고 싶을 때
- Cline이 모호함을 해결해야 할 때

### 워크스페이스 이름 규칙

워크스페이스 이름은 다음 기준으로 결정됩니다:
1. `.code-workspace` 파일의 `name` 필드(지정한 경우)
2. 폴더 이름(기본값)

동일한 이름의 폴더가 있으면 번호를 붙이거나 워크스페이스 파일에서 고유 이름을 지정하세요.

## 일반적인 구성

### 모노레포 개발

```
~/projects/my-app/
├── my-app.code-workspace      # 워크스페이스 설정 파일
├── web/          (React 프런트엔드)
├── api/          (Node.js 백엔드)  
├── mobile/       (React Native)
└── shared/       (공통 유틸리티)
```

모든 폴더가 하나의 Git 히스토리를 공유합니다. 패키지 간 변경이 원자적으로 처리됩니다.

**예시 프롬프트:** *"web과 mobile 앱 모두에서 새 백엔드 라우트를 사용하도록 API 엔드포인트를 업데이트해줘"*

### 마이크로서비스(개별 저장소)

```
~/projects/services/
├── services.code-workspace    # 워크스페이스 설정 파일
├── user-service/       (git: github.com/acme/user-service)
├── payment-service/    (git: github.com/acme/payment-service)
├── gateway/            (git: github.com/acme/api-gateway)
└── proto/              (git: github.com/acme/service-protos)
```

각 서비스는 자신의 저장소를 갖습니다. Cline은 proto 정의를 업데이트하고 모든 서비스에서 클라이언트를 재생성할 수 있습니다.

**예시 프롬프트:** *"proto의 UserProfile 메시지에 새 필드를 추가하고, user-service와 gateway가 이를 처리하도록 업데이트해줘"*

### 공유 계약 기반 풀스택

```
~/projects/fullstack/
├── fullstack.code-workspace   # 워크스페이스 설정 파일
├── client/         (git: github.com/acme/web-client)
├── server/         (git: github.com/acme/api-server)
└── types/          (git: github.com/acme/shared-types)
```

types 저장소는 클라이언트와 서버에서 사용하는 인터페이스를 정의합니다. 타입을 업데이트하면 Cline이 두 소비자를 모두 수정할 수 있습니다.

### 하이브리드 구성

```
~/projects/project/
├── project.code-workspace     # 워크스페이스 설정 파일
├── main-app/       (git: github.com/acme/main-app)
├── vendor/         (VCS 없음 - 벤더 의존성)
└── scripts/        (VCS 없음 - 로컬 자동화)
```

저장소와 일반 폴더를 혼합한 구성입니다. Cline은 각 폴더의 설정에 맞춰 동작합니다.

## 현재 제한사항

멀티 루트 워크스페이스 모드에서는 두 기능에 제한이 있습니다:

### Cline Rules

[Cline rules](/features/cline-rules) (`.clinerules/` 디렉터리)는 **기본 워크스페이스**(첫 번째 폴더)에서만 동작합니다. 다른 폴더의 규칙은 무시됩니다.

**우회 방법:** 공유 규칙은 기본 워크스페이스에 두거나, 모든 곳에 적용되는 전역 규칙(`~/Documents/Cline/Rules/`)을 사용하세요.

### 체크포인트

[체크포인트](/features/checkpoints)는 멀티 루트 워크스페이스 모드에서 비활성화됩니다. 이때 Cline은 경고를 표시합니다.

**이유:** 체크포인트는 변경 추적을 위해 섀도우 Git 저장소를 사용합니다. 여러 저장소에 걸쳐 체크포인트를 조정하는 것은 아직 지원되지 않는 복잡성이 있습니다.

**우회 방법:** 일반적인 Git 워크플로를 사용하세요. 자주 커밋하거나, 실험적 작업은 브랜치를 분리하세요.

두 제한사항은 단일 폴더 워크스페이스로 돌아오면 복원됩니다.

## 모범 사례

### 워크스페이스 구성

1. **연관된 프로젝트를 묶기**: 함께 변경이 필요한 프로젝트를 그룹화
2. **워크스페이스 파일 사용**: 팀 전체에서 재현 가능한 설정을 위해
3. **폴더 이름 명확화**: 워크스페이스 힌트가 직관적이도록
4. **기본 워크스페이스 고려**: Cline 규칙은 기본 폴더에 배치

### 효과적인 프롬프트 작성

- **중요할 때는 구체적으로**: *"backend 워크스페이스의 user 모델을 업데이트해줘"*
- **관계를 명시**: *"frontend는 contracts 워크스페이스의 타입을 사용한다"*
- **워크스페이스 간 변경 설명**: *"web과 mobile 둘 다 업데이트해야 한다"*
- **대규모 코드베이스에서 검색 범위 지정**: *"frontend 워크스페이스에서만 'TODO'를 검색해줘"*
