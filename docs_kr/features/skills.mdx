스킬은 특정 작업을 위한 Cline의 기능을 확장하는 모듈형 지침 세트입니다. 각 스킬은 상세 가이드, 워크플로, 선택적 리소스를 패키징하며, 요청과 관련 있을 때만 로드됩니다.

규칙(항상 활성)과 달리 스킬은 필요할 때만 로드됩니다. Cline은 실제로 필요해지기 전까지 스킬의 이름과 설명만 보므로, 컨텍스트나 성능에 영향을 주지 않고 수십 개의 스킬을 설치할 수 있습니다.

<Note>
스킬은 실험적 기능입니다. 설정 → Features → Enable Skills에서 활성화하세요.
</Note>

## 왜 스킬인가?

새로운 팀원을 온보딩한다고 생각해 보세요. 모든 문서를 한 번에 던지지 않고, 먼저 요약을 제공한 뒤 특정 작업을 할 때 세부 가이드를 안내합니다.

스킬도 같은 방식으로 동작합니다:
- **시작 시점**: Cline은 각 스킬의 간단한 설명만 봅니다
- **트리거 시점**: 해당 스킬의 전체 지침을 로드합니다
- **필요할 때만**: 추가 파일이 포함된 스킬은 참조될 때만 읽습니다

이 점진적 로딩 덕분에, 현재 작업과 무관한 정보를 컨텍스트 토큰으로 낭비하지 않으면서도 방대한 도메인 지식을 패키징할 수 있습니다.

## 스킬 만들기

각 스킬은 `SKILL.md` 파일이 포함된 디렉터리입니다:

```
my-skill/
├── SKILL.md          # 필수: 메인 지침
├── docs/             # 선택: 추가 문서
│   └── advanced.md
└── scripts/          # 선택: 유틸리티 스크립트
    └── helper.sh
```

`SKILL.md` 파일은 메타데이터와 지침 두 부분으로 구성됩니다.

```yaml
---
name: my-skill
description: Brief description of what this skill does and when to use it.
---

# My Skill

Detailed instructions for Cline to follow when this skill is activated.

## Steps

1. First, do this
2. Then do that
3. For advanced usage, see [advanced.md](docs/advanced.md)
```

**필수 필드:**
- `name`: 디렉터리 이름과 정확히 일치해야 합니다
- `description`: Cline이 이 스킬을 언제 사용할지 판단하는 설명(최대 1024자)

설명은 Cline이 스킬을 활성화할지 결정하는 기준이므로 매우 중요합니다. 스킬이 무엇을 하고, 어떤 상황에서 사용해야 하는지 구체적으로 작성하세요.

## 스킬 위치

스킬은 두 위치 중 하나에 저장할 수 있습니다:

**전역 스킬**은 모든 프로젝트에 적용됩니다:
- **macOS/Linux:** `~/.cline/skills/`
- **Windows:** `C:\Users\USERNAME\.cline\skills\`

**프로젝트 스킬**은 현재 워크스페이스에만 적용됩니다:
- `.cline/skills/` (권장)
- `.clinerules/skills/`
- `.claude/skills/` (Claude Code 호환)

전역 스킬과 프로젝트 스킬의 이름이 동일하면 전역 스킬이 우선합니다. 이를 통해 개인 워크플로에 맞게 커스터마이즈하면서도 프로젝트 기본값을 사용할 수 있습니다.

## 스킬 관리

채팅 입력 아래의 저울 아이콘을 클릭해 rules 및 workflows 패널을 엽니다. 스킬이 활성화되어 있으면 Skills 탭이 나타나며, 여기서 다음을 할 수 있습니다:

- 사용 가능한 모든 스킬 보기(전역 및 워크스페이스)
- 개별 스킬 활성/비활성 토글
- 템플릿으로 새 스킬 생성
- 필요 없는 스킬 삭제

스킬은 발견되면 기본적으로 활성화됩니다. 현재 프로젝트에서 사용하지 않더라도, 필요할 때 활성화하려면 토글로 끌 수 있습니다.

## Cline이 스킬을 사용하는 방식

메시지를 보내면 Cline은 사용 가능한 스킬 목록과 설명을 확인합니다. 요청이 어떤 스킬의 설명과 맞으면 `use_skill` 도구로 해당 스킬을 활성화해 전체 지침을 로드합니다.

예를 들어 AWS 배포 스킬이 있다면:

```yaml
---
name: aws-deploy
description: Deploy applications to AWS using CDK. Use when deploying, updating infrastructure, or managing AWS resources.
---
```

"이걸 AWS에 배포해줘"라고 요청하면 Cline이 스킬을 활성화하고, 상세 지침을 로드해 작업을 수행합니다.

## 예시: 데이터 분석 스킬

데이터 분석 작업을 위한 실용적인 스킬 예시입니다. `data-analysis/` 디렉터리를 만들고 다음 `SKILL.md`를 추가하세요:

```yaml
---
name: data-analysis
description: Analyze data files and generate insights. Use when working with CSV, Excel, or JSON data files that need exploration, cleaning, or visualization.
---
```

그리고 본문에 지침을 추가합니다:

````markdown
# Data Analysis

When analyzing data files, follow this workflow:

## 1. Understand the Data

- Read a sample of the file to understand its structure
- Identify column types and data quality issues
- Note any missing values or anomalies

## 2. Ask Clarifying Questions

Before diving in, ask the user:
- What specific insights are they looking for?
- Are there any known data quality issues?
- What format do they want for the output?

## 3. Perform Analysis

Use pandas for data manipulation:

```python
import pandas as pd

# Load and explore
df = pd.read_csv("data.csv")
print(df.head())
print(df.describe())
print(df.info())
```

For visualization, prefer matplotlib or seaborn depending on complexity.

## 4. Present Findings

- Start with a summary of key insights
- Support findings with specific numbers
- Include visualizations where they add clarity
- End with recommendations or next steps
````

## 보조 파일 묶기

스킬에는 필요할 때만 접근하는 추가 파일을 포함할 수 있습니다:

```
complex-skill/
├── SKILL.md
├── docs/
│   ├── setup.md
│   └── troubleshooting.md
├── templates/
│   └── config.yaml
└── scripts/
    └── validate.py
```

지침에서 이를 참조하세요:

````markdown
For initial setup, follow [setup.md](docs/setup.md).

Use the config template at `templates/config.yaml` as a starting point.

Run the validation script to check your configuration:
```bash
python scripts/validate.py
```
````

Cline은 지침에서 파일을 참조하면 `read_file`로 파일을 읽습니다. 스크립트는 직접 실행할 수 있으며, 스크립트 코드는 컨텍스트에 들어가지 않고 출력만 들어갑니다.

## 스킬 아이디어

스킬은 다음과 같은 작업에서 특히 빛납니다:
- 상세하고 다단계 워크플로가 필요한 작업
- 도메인 지식이나 모범 사례가 필요한 작업
- 매 대화마다 동일한 지침을 반복해야 하는 작업

예시:

- **릴리스 관리**: 버전 올리기, 변경 로그 생성, git 태깅, 배포
- **코드 리뷰**: 팀의 체크리스트와 품질 기준
- **DB 마이그레이션**: 롤백 절차를 포함한 안전한 스키마 진화
- **API 통합**: 특정 서드파티 서비스 연결과 에러 처리
- **문서화**: 선호하는 구조, 스타일 가이드, 도구
- **디버깅 워크플로**: 특정 유형의 이슈를 진단하는 체계적 접근
- **인프라**: 클라우드 환경을 위한 Terraform/CDK 패턴

최고의 스킬은 경험 많은 개발자의 머릿속에만 있던 조직 지식을 코드로 담아냅니다.

## 스킬 vs 규칙 vs 워크플로

| 기능 | 목적 | 활성 시점 |
|---------|---------|-------------|
| **Rules** | Cline의 동작 방식을 정의 | 항상(또는 컨텍스트에 따라) |
| **Workflows** | 단계별 작업 자동화 | `/workflow.md`로 호출 |
| **Skills** | 온디맨드 도메인 전문지식 | 요청과 매칭될 때 활성화 |

**Rules**는 제약과 선호를 설정합니다(예: "항상 TypeScript 사용", "이 스타일 가이드를 따르기").

**Workflows**는 특정 작업을 위해 명시적으로 호출하는 순서형 작업입니다(예: 릴리스 절차용 `/release.md`).

**Skills**는 관련 상황에서 자동으로 활성화되는 전문지식입니다(예: CSV 파일을 다룰 때 데이터 분석 스킬 활성화).

지속적인 제약에는 rules, 명시적 자동화에는 workflows, 필요할 때만 활성화되지만 항상 사용할 수 있어야 하는 도메인 지식에는 skills를 사용하세요.
