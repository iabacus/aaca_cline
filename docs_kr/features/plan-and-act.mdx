Plan & Act 모드는 구현 전에 충분히 계획을 세우는 구조적 AI 개발 접근을 나타냅니다. 이 이중 모드 시스템은 개발자가 더 유지보수 가능하고 정확한 코드를 만들도록 돕고, 반복 시간을 줄여줍니다.

#### Plan 모드: 먼저 생각하기

Plan 모드는 무엇을 만들지, 어떻게 만들지를 Cline과 함께 정리하는 단계입니다. 이 모드에서 Cline은:

-   전체 코드베이스를 읽어 컨텍스트를 이해할 수 있음
-   파일을 변경하지 않음
-   요구사항 이해와 전략 수립에 집중
-   코드 작성 전에 잠재 이슈를 식별하도록 도움

<Tip>
Plan 모드에서는 [받아쓰기](/features/dictation)를 활용해 보세요. 복잡한 요구사항을 타이핑하는 대신 자연스럽게 말로 전달하면 됩니다. 빠른 왕복 계획 논의에 딱 맞습니다.
</Tip>

#### Act 모드: 구현하기

계획이 준비되면 Act 모드로 전환합니다. 이때 Cline은:

-   모든 구현 기능을 사용할 수 있음
-   코드베이스를 변경할 수 있음
-   계획 세션의 맥락을 그대로 기억함
-   함께 수립한 전략을 실행함

### 워크플로 가이드

새 기능을 만들거나 복잡한 버그를 수정할 때, 다음 흐름이 효과적입니다:

1. Plan 모드에서 Cline에게 만들고 싶은 것을 설명한다
2. Cline이 관련 파일을 보며 코드베이스를 탐색한다
3. 엣지 케이스와 잠재 이슈를 고려해 최적의 접근을 함께 찾는다
4. 계획에 확신이 생기면 Act 모드로 전환한다
5. Cline이 계획에 따라 솔루션을 구현한다

중요한 개발 작업은 Plan 모드에서 시작하세요:

Act 모드에서 Cline은:

-   합의된 계획대로 실행
-   코드베이스 변경
-   계획 단계의 컨텍스트 유지

#### 3. 필요 시 반복

복잡한 프로젝트는 여러 번의 Plan-Act 사이클이 필요합니다:

-   예상치 못한 복잡성이 등장하면 Plan 모드로 복귀
-   해결책 구현은 Act 모드에서 수행
-   품질을 확보하면서 개발 흐름을 유지

### 모범 사례

#### 계획 단계

1. 요구사항을 포괄적으로 작성
2. 관련 컨텍스트를 처음부터 공유
3. Cline이 읽지 않은 관련 파일을 안내
4. 구현 전에 접근 방식 검증

#### 구현 단계

1. 수립한 계획을 따르기
2. 목표 대비 진행 상황 모니터링
3. 변경 사항과 영향 추적
4. 중요한 의사결정 문서화

### 고급 사용자 팁

#### 계획 강화

-   구현 전에 엣지 케이스를 탐색하는 데 Plan 모드 사용
-   예상치 못한 복잡성이 나오면 Plan 모드로 돌아가기
-   [파일 읽기](/features/at-mentions/file-mentions)를 활용해 초기에 가정을 검증
-   계획을 마크다운 파일로 작성해 참고 자료로 남기기

### 공통 패턴

#### 각 모드를 언제 사용할까

Plan 모드는 다음 상황에서 특히 효과적입니다:

-   접근 방식이 명확하지 않은 새 작업을 시작할 때
-   문제가 어디 있는지 확신할 수 없는 까다로운 버그를 디버깅할 때
-   코드베이스 여러 부분에 영향을 주는 아키텍처 결정을 내릴 때
-   복잡한 워크플로나 기능을 이해하려 할 때

Act 모드는 다음 상황에 적합합니다:

-   이미 계획된 해결책을 구현할 때
-   접근 방식이 분명한 일상적 변경
-   코드베이스에 확립된 패턴을 따를 때
-   테스트 실행과 사소한 조정을 할 때
