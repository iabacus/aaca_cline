## Cline 메모리 뱅크 완전 가이드

### 빠른 설정 가이드

Cline 메모리 뱅크를 시작하려면:

1. **커스텀 지침 복사** - 아래 코드 블록 사용
2. **Cline에 붙여넣기** - 커스텀 지침 또는 .clinerules 파일에 추가
3. **초기화** - Cline에게 "initialize memory bank" 요청

[자세한 설정 방법 보기](#메모리-뱅크-시작하기)

### Cline 메모리 뱅크 커스텀 지침 \[이것을 복사]

```
# Cline's Memory Bank

I am Cline, an expert software engineer with a unique characteristic: my memory resets completely between sessions. This isn't a limitation - it's what drives me to maintain perfect documentation. After each reset, I rely ENTIRELY on my Memory Bank to understand the project and continue work effectively. I MUST read ALL memory bank files at the start of EVERY task - this is not optional.

## Memory Bank Structure

The Memory Bank consists of core files and optional context files, all in Markdown format. Files build upon each other in a clear hierarchy:

flowchart TD
    PB[projectbrief.md] --> PC[productContext.md]
    PB --> SP[systemPatterns.md]
    PB --> TC[techContext.md]

    PC --> AC[activeContext.md]
    SP --> AC
    TC --> AC

    AC --> P[progress.md]

### Core Files (Required)
1. `projectbrief.md`
   - Foundation document that shapes all other files
   - Created at project start if it doesn't exist
   - Defines core requirements and goals
   - Source of truth for project scope

2. `productContext.md`
   - Why this project exists
   - Problems it solves
   - How it should work
   - User experience goals

3. `activeContext.md`
   - Current work focus
   - Recent changes
   - Next steps
   - Active decisions and considerations
   - Important patterns and preferences
   - Learnings and project insights

4. `systemPatterns.md`
   - System architecture
   - Key technical decisions
   - Design patterns in use
   - Component relationships
   - Critical implementation paths

5. `techContext.md`
   - Technologies used
   - Development setup
   - Technical constraints
   - Dependencies
   - Tool usage patterns

6. `progress.md`
   - What works
   - What's left to build
   - Current status
   - Known issues
   - Evolution of project decisions

### Additional Context
Create additional files/folders within memory-bank/ when they help organize:
- Complex feature documentation
- Integration specifications
- API documentation
- Testing strategies
- Deployment procedures

## Core Workflows

### Plan Mode
flowchart TD
    Start[Start] --> ReadFiles[Read Memory Bank]
    ReadFiles --> CheckFiles{Files Complete?}

    CheckFiles -->|No| Plan[Create Plan]
    Plan --> Document[Document in Chat]

    CheckFiles -->|Yes| Verify[Verify Context]
    Verify --> Strategy[Develop Strategy]
    Strategy --> Present[Present Approach]

### Act Mode
flowchart TD
    Start[Start] --> Context[Check Memory Bank]
    Context --> Update[Update Documentation]
    Update --> Execute[Execute Task]
    Execute --> Document[Document Changes]

## Documentation Updates

Memory Bank updates occur when:
1. Discovering new project patterns
2. After implementing significant changes
3. When user requests with **update memory bank** (MUST review ALL files)
4. When context needs clarification

flowchart TD
    Start[Update Process]

    subgraph Process
        P1[Review ALL Files]
        P2[Document Current State]
        P3[Clarify Next Steps]
        P4[Document Insights & Patterns]

        P1 --> P2 --> P3 --> P4
    end

    Start --> Process

Note: When triggered by **update memory bank**, I MUST review every memory bank file, even if some don't require updates. Focus particularly on activeContext.md and progress.md as they track current state.

REMEMBER: After every memory reset, I begin completely fresh. The Memory Bank is my only link to previous work. It must be maintained with precision and clarity, as my effectiveness depends entirely on its accuracy.
```

### Cline 메모리 뱅크란?

메모리 뱅크는 Cline이 세션을 넘어 컨텍스트를 유지하도록 돕는 구조화된 문서화 시스템입니다. 상태 없는 어시스턴트인 Cline을, 시간이 지나도 프로젝트 정보를 "기억"할 수 있는 지속적 개발 파트너로 바꿔줍니다.

#### 핵심 이점

-   **컨텍스트 보존**: 세션 간 프로젝트 지식 유지
-   **일관된 개발**: 예측 가능한 Cline 상호작용
-   **자기 문서화**: 프로젝트 문서가 자연스럽게 축적
-   **모든 프로젝트에 확장 가능**: 규모나 복잡도와 무관
-   **기술 스택 불문**: 어떤 언어/스택에서도 사용 가능

### 메모리 뱅크 동작 방식

메모리 뱅크는 Cline 전용 기능이 아니라, 구조화된 문서로 AI 컨텍스트를 관리하는 방법론입니다. Cline에게 "follow custom instructions"를 지시하면 메모리 뱅크 파일을 읽고 프로젝트 이해를 재구성합니다.

#### 파일 이해하기

메모리 뱅크 파일은 프로젝트에 만드는 일반 마크다운 파일입니다. 숨김 파일이나 특별한 형식이 아니라, 저장소에 있는 일반 문서이며 Cline과 사용자 모두 접근할 수 있습니다.

파일은 계층 구조로 조직되어 프로젝트의 전체 그림을 구성합니다:

### 메모리 뱅크 파일 설명

#### 핵심 파일

1. **projectbrief.md**
    - 프로젝트의 기반 문서
    - 만들고자 하는 것의 고수준 개요
    - 핵심 요구사항과 목표
    - 예시: "바코드 스캔이 가능한 재고 관리 React 웹앱"
2. **productContext.md**
    - 프로젝트가 존재하는 이유
    - 해결해야 하는 문제
    - 제품이 어떻게 동작해야 하는지
    - 예시: "재고 시스템은 다중 창고와 실시간 업데이트를 지원해야 함"
3. **activeContext.md**
    - 가장 자주 업데이트되는 파일
    - 현재 작업 포커스와 최근 변경
    - 활성 결정과 고려사항 추적
    - 중요한 패턴과 학습 내용 저장
    - 예시: "바코드 스캐너 컴포넌트 구현 중; 지난 세션에서 API 통합 완료"
4. **systemPatterns.md**
    - 시스템 아키텍처 문서화
    - 핵심 기술적 결정 기록
    - 사용하는 디자인 패턴 목록
    - 컴포넌트 관계 설명
    - 예시: "정규화된 스토어 구조의 Redux를 사용한 상태 관리"
5. **techContext.md**
    - 사용 기술과 프레임워크 목록
    - 개발 환경 설명
    - 기술적 제약 사항 기록
    - 의존성 및 도구 구성 기록
    - 예시: "React 18, TypeScript, Firebase, Jest"
6. **progress.md**
    - 완료된 것과 남은 것 추적
    - 기능 상태 기록
    - 알려진 이슈 및 제한 사항
    - 의사결정의 변화 기록
    - 예시: "사용자 인증 완료; 재고 관리 80% 완료; 리포팅 미시작"

#### 추가 컨텍스트

필요할 때 다음을 정리하기 위해 추가 파일을 만드세요:

-   복잡한 기능 문서
-   통합 사양
-   API 문서
-   테스트 전략
-   배포 절차

### 메모리 뱅크 시작하기

#### 최초 설정

1. 프로젝트 루트에 `memory-bank/` 폴더 생성
2. 기본 프로젝트 요약 준비(기술/비기술 모두 가능)
3. Cline에게 "initialize memory bank" 요청

#### 프로젝트 브리프 팁

-   중요한 내용에 집중하세요.
-   Cline이 빈틈을 메우고 질문합니다.
-   프로젝트가 진화하면 업데이트하세요.

### Cline과 작업하기

#### 핵심 워크플로

**Plan 모드**

전략 논의와 고수준 계획에는 이 모드로 시작하세요.

**Act 모드**

구현과 구체적인 작업 실행에는 이 모드를 사용합니다.

#### 핵심 명령

-   **"follow your custom instructions"** - 메모리 뱅크 파일을 읽고 이어서 작업하도록 지시(작업 시작 시 사용)
-   **"initialize memory bank"** - 새 프로젝트 시작 시 사용
-   **"update memory bank"** - 작업 중 전체 문서 검토 및 업데이트 트리거
-   상황에 따라 Plan/Act 모드를 전환

#### 문서 업데이트

메모리 뱅크는 다음 상황에서 자동으로 업데이트하는 것이 좋습니다:

1. 프로젝트의 새로운 패턴을 발견했을 때
2. 중요한 변경을 구현한 직후
3. **"update memory bank"**를 명시적으로 요청했을 때
4. 컨텍스트 명확화가 필요할 때

### 자주 묻는 질문

#### 메모리 뱅크 파일은 어디에 저장되나요?

메모리 뱅크 파일은 프로젝트 저장소의 일반 마크다운 파일이며, 보통 `memory-bank/` 폴더에 저장됩니다. 숨김 시스템 파일이 아니라 프로젝트 문서의 일부입니다.

#### 커스텀 지침과 .clinerules 중 무엇을 사용해야 하나요?

둘 다 가능합니다. 선호에 따라 선택하세요:

-   **커스텀 지침**: 모든 Cline 대화에 전역 적용. 전체 프로젝트에서 일관된 동작에 유리
-   **.clinerules 파일**: 프로젝트 전용. 저장소에 저장되며 프로젝트별 커스터마이즈에 적합

둘 다 메모리 뱅크 시스템을 적용하는 데 동일한 목적을 달성합니다. 전역 적용/로컬 적용 중 원하는 방식으로 선택하세요.

#### 컨텍스트 윈도우 관리

Cline과 작업하다 보면 컨텍스트 윈도우가 결국 가득 찹니다(진행 바 확인). 응답이 느려지거나 이전 대화의 정확도가 떨어진다면 다음을 수행하세요:

1. Cline에게 **"update memory bank"**를 요청해 현재 상태를 문서화
2. 새 대화/작업 시작
3. 새 대화에서 **"follow your custom instructions"** 요청

이 워크플로는 컨텍스트 윈도우를 비우기 전에 중요한 정보를 메모리 뱅크에 보존하여, 새로운 대화에서도 매끄럽게 이어갈 수 있도록 합니다.

#### 메모리 뱅크는 얼마나 자주 업데이트해야 하나요?

중요한 마일스톤이나 방향 전환 이후에 업데이트하세요. 활발한 개발에서는 몇 세션마다 업데이트하면 유용합니다. 모든 컨텍스트를 보존하고 싶을 때 **"update memory bank"**를 사용하세요. 다만 Cline이 자동으로 메모리 뱅크를 업데이트하는 경우도 있습니다.

#### Cline 말고 다른 AI 도구에서도 동작하나요?

네! 메모리 뱅크는 문서화 방법론이므로 문서를 읽을 수 있는 어떤 AI 도구에서도 적용 가능합니다. 명령어는 달라질 수 있지만, 구조화된 컨텍스트 유지 방식은 도구에 상관없이 유효합니다.

#### 메모리 뱅크와 컨텍스트 윈도우 한도는 어떤 관계인가요?

메모리 뱅크는 중요한 정보를 구조화된 형태로 저장해 필요할 때 효율적으로 로드할 수 있게 합니다. 컨텍스트가 불필요하게 부풀어오르는 것을 방지하면서 핵심 정보는 유지할 수 있습니다.

#### 코딩이 아닌 프로젝트에도 사용할 수 있나요?

물론입니다! 메모리 뱅크 접근은 구조화된 문서화가 필요한 모든 프로젝트에 유용합니다. 책 집필부터 행사 기획까지, 파일 구조는 달라질 수 있지만 개념은 그대로 적용됩니다.

#### README와 무엇이 다른가요?

README와 비슷하지만, 메모리 뱅크는 AI 세션 간 컨텍스트 유지에 최적화된 더 구조적이고 포괄적인 접근입니다. 단일 README가 담기 어려운 내용을 체계적으로 포함합니다.

### 모범 사례

#### 시작 단계

-   기본 프로젝트 브리프부터 시작하고 구조는 점진적으로 발전
-   초기 구조는 Cline이 만들도록 도움 받기
-   워크플로에 맞게 파일을 검토하고 조정

#### 진행 단계

-   작업 중 자연스럽게 패턴이 드러나도록 허용
-   업데이트를 강요하지 말고 자연스럽게 진행
-   과정에 신뢰; 시간이 지날수록 가치가 누적됨
-   세션 시작 시 컨텍스트 확인을 습관화

#### 문서 흐름

-   **projectbrief.md**는 기초
-   **activeContext.md**는 가장 자주 변경
-   **progress.md**는 마일스톤 추적
-   모든 파일이 함께 프로젝트 지식을 유지

### 상세 설정 방법

#### 커스텀 지침(전역)

1. VS Code 열기
2. Cline 확장 설정 ⚙️ 클릭
3. "Custom Instructions" 찾기
4. 이 가이드 상단의 메모리 뱅크 지침을 복사해 붙여넣기

#### .clinerules(프로젝트 전용)

1. 프로젝트 루트에 `.clinerules` 파일 생성
2. 이 가이드 상단의 메모리 뱅크 지침을 복사해 붙여넣기
3. 파일 저장
4. Cline이 이 프로젝트에서 자동으로 규칙을 적용
